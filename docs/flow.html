team-sync · architecture

Real-time, zero-drift workspace for AI coding agents

How developer intent (OKF over NATS) and codebase structure (Graphify) are kept in lock-step across a team of agents — even when a laptop goes offline. Supervised by pm2; isolated per project.

01The pieces

Two sources of truth, one bus, one bridge — split across a machine tier and a project tier.

Central NATS Hubteam

Cloud broker. JetStream keeps a per-project stream (TEAM_SYNC_<PROJECT>) and replays what an offline agent missed. nats/hub.conf

Local Leaf Nodemachine

One per machine, on 127.0.0.1:4222. Dumb transport: acks instantly, buffers offline, syncs up later. Knows nothing about projects. generated leaf.conf

Sync Daemonproject

One per enrolled project. Bound to that repo's subjects + .context/. Writes inbound events and refreshes Graphify. tooling/sync-daemon.js

Graphifyproject

Tree-sitter dependency graph, queried by agents over MCP. Local & per-machine: graphify-out/graph.json. external CLI

OKF + team-sync.jsonproject

OKF carries each change as a log_event. The committed team-sync.json marker is what opts a project in. docs/OKF.md

Supervisormachine

Harness-bound (default): session hooks start/stop the leaf + daemon, ref-counted, only while an agent codes. Or pm2 for always-on. session.js · pm2

02The whole picture

One developer machine: a shared leaf node, two enrolled projects on their own subjects, one private project that never broadcasts.

Project identity lives in the subject (team.sync.<project>.events.*) and the stream (TEAM_SYNC_<PROJECT>), both derived deterministically from team-sync.json — not in the leaf node, which is shared and project-agnostic. Switching projects needs no mode switch: each daemon is permanently bound to its repo, and the publisher picks context from its working directory.

03The flows

Click through each lifecycle we implemented.

No service to install, no pm2. In the default harness-bound mode the leaf + daemon are started by the AI session itself and stop when it closes — you just commit a marker and open a session.

  1. Once per project · team projectsMark the project.

    Commit a team-sync.json. Its presence + broadcast:true is the opt-in; private projects simply omit it.

    { "project": "acme-api", "broadcast": true,
  "hub": "nats://hub.acme.internal:7422" }
  2. Once per cloneInstall git hooks.

    Keeps the local Graphify graph fresh after commits/pulls. That's the whole install step.

    npm install && npm run hooks:install
  3. Every time you codeOpen a session.

    Claude Code's committed .claude/settings.json runs session.js up on start (spawns leaf + daemon detached, ref-counted) and down on close. Trust the hooks once, and you're synced — nothing runs between sessions.

    # just open the project in your harness
# (other harnesses: wire up/down — see docs/HARNESS.md)
  4. AnytimeInspect. 
    npm run session:status   # active sessions + leaf/daemon pids
npm run session:gc       # reap a crashed session

Optional — always-on (pm2) for unattended machines. Use this only when there's no interactive harness session to bind the lifecycle to and you need sync running 24/7: a shared dev box, a build / CI agent, a harness with no session hooks, or a machine that should keep syncing across reboots. Install with npm run setup:machine then npm run enroll (per project); manage with npm run ts:list / npm run leave. See docs/SETUP.md. Pick one mode per machine — never pm2 and harness-bound for the same project.

  1. Agent · AGENTS.md §2Query the structure.

    Before editing, ask Graphify what depends on the target file — over MCP, or on the CLI.

    graphify explain "src/auth/session.ts"
graphify path "LoginForm" "SessionStore"
  2. Agent · AGENTS.md §1–2Read peer intent.

    Scan .context/live-logs/ for recent deltas. If a peer is mid-refactor on your target, you are prohibited from touching it until that log resolves.

  3. AgentCheck the rules.

    Confirm the change doesn't violate CLAUDE.md / AGENTS.md, then write code.

  1. Agent · AGENTS.md §3AAuthor the OKF payload.

    Write a log_event to a temp file using the template.

    .context/temp-event.md   # type/title/author/timestamp/impacted_files/breaking
  2. Agent → Daemon (publish mode)Broadcast.

    The publisher checks the project is enrolled, validates the OKF schema, then JetStream-publishes to team.sync.<project>.events.<agentId> with a msgID for dedupe. A private project is refused here.

    node tooling/sync-daemon.js --publish .context/temp-event.md
  3. Leaf → Hub → peer leaf nodesFan out.

    The event lands in this project's TEAM_SYNC_<PROJECT> stream on the hub and is delivered to every peer's per-project durable consumer.

  4. Peer Daemon (watch mode)Materialise + refresh.

    Each peer's daemon for that project writes the event to .context/live-logs/<ts>-<slug>.md, acks it, and triggers a debounced graph re-extraction. Self-published events are skipped to avoid echo.

    graphify update .
  5. AgentClean up.

    Delete the temp file and report success. Peers now see the change in both their live-logs and their graph.

  1. Leaf node · offlineLaptop loses connectivity.

    Agents keep broadcasting to localhost:4222. The shared leaf node acks instantly and buffers locally — no hang, for any enrolled project.

  2. JetStream · hubPeers' events accumulate.

    While you're dark, the hub keeps appending each project's deltas to its TEAM_SYNC_<PROJECT> stream. Nothing is lost.

  3. Leaf ↔ HubReconnect.

    The daemon connects with maxReconnectAttempts: -1; buffered outbound events flush up, and each project's durable consumer (DeliverPolicy.All, keyed per machine + project) replays everything missed.

  4. DaemonCatch up on demand.

    Drain missed events without starting the long-running watcher:

    npm run catchup        # node tooling/sync-daemon.js --once
  1. WhyThe graph is local & per-machine.

    Any git op that changes the working tree desyncs graphify-out/graph.json from the code on disk. A query against a stale graph gives wrong dependency answers.

  2. Install once per cloneEnable the hooks. 
    npm run hooks:install   # sh tooling/git-hooks/install.sh

    Writes thin stubs into .git/hooks/ that exec the tracked scripts in tooling/git-hooks/.

  3. Automatic thereafterRe-extract on every mutation.

    The hooks run graphify update . after:

    post-commit     → your own edits
post-merge      → git pull (fetch + merge)
post-rewrite    → git rebase / commit --amend
post-checkout   → branch switch (flag = 1 only)
  4. AlternativeOr watch continuously.

    Running graphify watch . as a background daemon covers all of the above, rebuilding on any file change.

Harness-bound mode binds the leaf + daemon to your AI session, so nothing runs when you're not coding. Handles the messy realities of hooks — blocking, ref-counting, and crashes. tooling/session/session.js · .claude/settings.json.

  1. Harness · SessionStart → upOpen a session.

    The hook runs session.js up, reading session_id + cwd from stdin JSON. A private project (no team-sync.json / broadcast:false) is a silent no-op.

    node tooling/session/session.js up --harness claude
  2. Launcher · detached + ref-countedSpawn, don't block.

    Writes one token file per session under ~/.team-sync/sessions/, then spawns the leaf + daemon detached and exits 0 immediately — SessionStart blocks, so it must return fast. Second session on the same machine just adds a token; the shared leaf is reused.

  3. ReconcileDesired = union of live sessions.

    One leaf for the union of active projects' hubs; one daemon per active project. Idempotent — re-running up changes nothing if already correct.

  4. Harness · SessionEnd → downClose a session.

    Removes that session's token and reconciles. The leaf + a project's daemon stop only when their last token is gone. reason:"clear" (Claude /clear) is ignored — the process keeps running.

    node tooling/session/session.js down --harness claude
  5. Crash · no SessionEndSelf-healing GC.

    If a harness is killed, no down fires. Every command first GCs tokens whose owning pid is dead (or older than SYNC_SESSION_TTL_MS), so the next up/gc reaps the zombie and tears down. Add a periodic session.js gc as a safety net.

    npm run session:status   # active sessions + leaf/daemon pids
npm run session:gc       # reap dead sessions, reconcile

04The OKF log_event

Validated by tooling/okf.js on publish and again on receive.

FieldTypeRole
typestringMust equal log_event.
titlestringHuman summary; slugified into the live-log filename.
authorstringAgentIdentity/HumanOwnerName.
timestampISO-8601Must parse via Date.parse.
impacted_fileslistFiles touched — logged, and a signal to re-extract the graph.
breakingbooleantrue surfaces a [BREAKING] marker in the daemon log.

Subject & stream are derived from team-sync.json's project — e.g. "Acme API" → subject team.sync.acme-api.events.*, stream TEAM_SYNC_ACME_API. Deterministic, so every teammate agrees without shared mutable config.

Implementation note: the daemon runs graphify update . over the whole tree per event (the CLI re-extracts itself), so impacted_files currently drives logging, not a targeted rebuild.
